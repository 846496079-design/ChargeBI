## 01 逻辑起点

标题：

> 逻辑起点：让业务人员直接从数据库中找答案

文案：

> 传统 BI 看板能回答固定问题，但面对临时问题时，业务人员仍然需要找研发或数据分析师取数。ChargeBI 的核心目标，是让非技术人员可以像问数据分析师一样，用自然语言直接查询数据库，并获得可解释的业务结论。

问题对比：

| 传统方式 | ChargeBI |
|---|---|
| 找固定看板 | 直接自然语言提问 |
| 找研发写 SQL | AI 自动匹配 Schema 并生成 SQL |
| 只看到数字 | 同时获得结论、图表和追问建议 |
| 临时问题响应慢 | 多轮对话连续分析 |

## 02 为什么选择充电运营

标题：

> 为什么选城市充电网络？

文案：

> 我最初在智慧城市、工业供应链、美食出行、园区能碳等方向中做过比较。最终选择充电运营，是因为它同时满足三个条件：数据结构适合构建、用户一眼能理解、业务指标适合问数。

判断矩阵：

| 维度 | 选择理由 |
|---|---|
| 数据可构建 | 站点、充电桩、订单、故障、运维、天气、节假日都能形成结构化数据库 |
| 场景好理解 | 地图上看到站点分布，用户很容易理解“问城市充电网络” |
| 问数适配 | 收入、订单、利用率、故障率、峰谷时段、区域排名都适合自然语言查询 |
| 展示有亮点 | 新能源、智慧城市、地图、AI Copilot 组合后更适合作品集展示 |

## 03 产品目标

标题：

> 产品目标：不是聊天框，而是完整问数链路

文案：

> ChargeBI 不把 AI 问数简化成一个聊天框。一个可信的问数产品，需要让用户知道系统如何理解问题、匹配了哪些数据、生成了什么查询、结果是否可靠，以及下一步还能追问什么。

核心链路：

1. 问题理解
2. 时间、区域、指标识别
3. Schema 匹配
4. SQL 生成
5. SQL 安全校验
6. 查询执行
7. 结果解释
8. 图表展示
9. 多轮追问
10. 错误与权限引导

## 04 核心体验

标题：

> 核心体验：左侧问数，右侧地图

文案：

> ChargeBI 的第一屏采用双栏工作台。左侧是 AI 问数区，承载对话、推荐问题、分析结论、图表和可信解释；右侧是上海都市圈示例充电网络地图，展示站点分布、状态、区域聚合和运营指标。

关键体验：

- 进入页面即可看到地图和推荐问题
- 用户可以自由输入，也可以从推荐问题开始
- 查询结果会联动地图高亮相关区域或站点
- 每次回答后提供追问建议
- 技术细节默认折叠，用户需要时可以查看 Schema 和 SQL

## 06 AI 产品设计

标题：

> AI 产品设计：开放输入 + 产品化护栏

文案：

> 开放式输入能让 Demo 更像真实产品，但也会带来不稳定。我的策略不是完全信任模型，而是在 AI 链路中加入产品化护栏：白名单 Schema、SQL 只读校验、敏感信息拦截、查询模板兜底和错误引导。

护栏设计：

- 只允许查询白名单表
- 只允许 SELECT
- 拦截手机号、车牌等敏感问题
- 超范围问题不进入 SQL 生成
- 问题有歧义时先澄清
- SQL 失败时解释原因并推荐相近问题

## 07 可展示能力

标题：

> Demo 能展示什么？

推荐展示问题：

1. 最近 30 天各区域充电收入排名如何？
2. 哪些站点利用率最低，可能需要优化？
3. 上周故障率最高的 5 个站点是哪些？
4. 浦东新区和徐汇区最近 30 天订单量有什么差异？
5. 为什么本月订单量下降？
6. 未来 7 天哪些站点可能出现高峰压力？
7. 这些站点的收入损失是多少？
8. 查询某个用户手机号。

第 8 个问题用于展示权限和隐私保护。
