# 页面原型结构与交互流程 v1.0

## 设计目标

ChargeBI 的产品界面需要做到：

- 一眼看懂这是城市充电网络运营分析产品
- 一眼知道可以直接提问
- 支持地图、指标、问答、图表和可信解释同时协作
- 展示 AI 问数链路，而不是只有聊天回复
- 与个人网站视觉风格一致，但进入 Demo 后呈现完整产品本身

## 页面结构

第一版只做一个核心产品页：

- 页面名称：ChargeBI 工作台
- 路由建议：`/chargebi` 或独立部署入口
- 布局：左侧 AI 问数工作台，右侧地图和运营指标

## 整体布局

推荐桌面端布局：

| 区域 | 宽度 | 内容 |
|---|---:|---|
| 顶部导航 | 100% | 产品名、数据说明、全局时间范围、Demo 标识 |
| 左侧工作台 | 44% | 对话、推荐问题、结果、图表、表格、可信解释 |
| 右侧地图区 | 56% | 地图、指标卡、区域/站点详情、筛选 |

桌面端高度策略：

- 左右两栏高度固定为首屏工作台高度。
- 地图区域不能被对话内容撑高。
- 对话消息流在左侧工作台内部滚动。
- 地图和指标区在右侧工作台内部保持稳定比例。
- 查询结果增多时，页面整体不应无限拉长。

移动端可降级为上下布局：

1. 顶部导航
2. 地图与指标概览
3. 问数工作台
4. 结果与可信解释

### 内容

- 副标题：智能充电运营问数助手
- 数据范围：上海都市圈示例数据
- 当前时间范围：最近 12 个月
- 入口按钮：重置会话
- 可选按钮：查看数据说明

### 文案建议

主标题：

> ChargeBI

副标题：

> 智能充电运营问数助手

数据说明短文案：

> 基于公开数据结构与模拟运营数据构建的作品集 Demo

## 左侧 AI 问数工作台

### 模块顺序

1. 问候与场景引导
2. 推荐问题
3. 对话消息流
4. 用户输入框
5. 回答结果区
6. 可信解释折叠区

### 空状态

首次进入时，左侧不应空白。

建议展示：

> 你好，我是 ChargeBI。你可以直接问我上海都市圈充电网络的收入、订单、利用率、故障、节假日表现和运营建议。

推荐问题：

- 最近 30 天各区域充电收入排名如何？
- 哪些站点利用率最低，可能需要优化？
- 上周故障率最高的 5 个站点是哪些？
- 未来 7 天哪些站点可能出现高峰压力？

### 输入框

输入框占底部固定区域。

能力要求：

- 支持自然语言输入
- 支持回车提交
- 支持多轮追问
- 支持点击推荐问题直接填入并提交
- 支持加载中禁用重复提交

占位文案：

> 试试问：浦东和徐汇最近 30 天哪个区域利用率更高？

### 推荐问题策略

推荐问题分为三类：

1. 初始推荐：帮助用户上手
2. 结果后追问：基于当前查询结果生成
3. 错误后推荐：当问题超范围或失败时引导修正

## 右侧地图和指标区

### 地图

地图用于建立场景理解，不承担复杂 GIS 功能。

第一版地图能力：

- 展示上海都市圈示例区域
- 展示充电站点点位
- 点位颜色表示站点状态
- 点位大小表示收入或订单规模
- 支持区域聚合
- 支持点击区域或站点
- 支持查询结果后高亮相关区域或站点

地图视觉要求：

- 地图必须保持固定视窗比例，避免被内容撑成长条。
- 点位和区域标签需要在默认视图中清晰可见。
- 第一版改为 Leaflet 真实地图底图，使用站点经纬度落点。
- 如果点位过多影响阅读，默认展示重点站点，查询结果再高亮相关站点。
- 如果真实底图加载失败，应保留站点详情和指标卡，不影响问数主链路。
- 充电站点位需要有高对比度外圈、图标符号和状态颜色，不能只依赖小圆点。
- 站点详情卡只在点击站点后出现；点击地图空白处时关闭详情卡。

点位状态建议：

| 状态 | 颜色 | 含义 |
|---|---|---|
| 正常 | 蓝绿色 | 运营稳定 |
| 高负荷 | 暖橙 | 利用率较高 |
| 低利用 | 灰蓝 | 利用率偏低 |
| 故障/离线 | 红色 | 需要关注 |
| 维护中 | 紫灰 | 运维处理中 |

### 指标卡

地图上方或右侧展示 4 到 6 个关键指标：

- 今日订单量
- 最近 30 天收入
- 平均利用率
- 故障站点数
- 高负荷站点数
- 站点健康度

指标卡需要支持随筛选变化而更新。

### 地图点击交互

点击区域后：

- 地图高亮该区域
- 地图自动缩放到该区域范围
- 地图绘制该区域的边界框，帮助用户理解空间范围
- 右侧或地图下方出现区域摘要
- 左侧输入框上方展示上下文提示

示例：

> 已选择：浦东新区。你可以继续问“这个区域最近 30 天收入趋势如何？”

点击站点后：

- 地图高亮站点
- 展示站点详情卡
- 左侧上下文记录当前站点

站点详情卡字段：

- 站点名称
- 区域
- 站点类型
- 当前状态
- 今日订单量
- 最近 30 天收入
- 利用率
- 故障次数

### 结果与地图联动

当问数结果包含站点列表时：

- 地图高亮结果中的相关站点。
- 非相关站点淡显，降低视觉干扰。
- 如果用户点击结果表格中的某一行站点，地图只高亮该站点。
- 如果用户点击图表中的某个站点柱子，地图只高亮该站点。
- 点击地图空白处，取消单点聚焦，但保留当前查询结果高亮。

区域快捷按钮交互：

- 需要提供“全区域”按钮，用于退出单一区域筛选，恢复全局地图视角。
- 点击区域按钮后，地图 `fitBounds` 到该区域真实 GeoJSON 边界。
- 用醒目的暖橙色边界高亮区域范围。
- 再点击其他区域按钮时，替换旧边界框。
- 快捷按钮只展示行政区，避免业务区与行政区重叠造成歧义。
- 选中行政区后，区内站点保持正常显示，区外站点淡显。

## 查询流程

用户提交问题后，系统进入 AI 工作流。

### 工作流状态

在回答生成过程中展示：

1. 理解问题
2. 识别指标、时间和区域
3. 匹配数据表和字段
4. 生成 SQL
5. 校验查询安全性
6. 执行查询
7. 生成分析结论

每个状态应以简洁进度条或步骤列表展示。

### 问题理解展示

回答前或回答上方展示：

> 我理解你的问题是：查询最近 30 天上海各区域的充电收入，并按收入从高到低排序。

结构化理解：

| 项目 | 示例 |
|---|---|
| 分析意图 | 区域收入排名 |
| 时间范围 | 最近 30 天 |
| 指标 | 充电收入 |
| 维度 | 区域 |
| 筛选条件 | 上海都市圈 |

## 回答结果结构

一次完整回答包含：

1. 简短结论
2. 关键指标
3. 图表
4. 明细表格
5. 地图联动说明
6. 追问建议
7. 可信解释折叠区

### 简短结论

结论应像业务分析师，而不是数据库输出。

示例：

> 最近 30 天，浦东新区充电收入最高，占总收入的 28.4%。收入主要由陆家嘴、张江和世纪公园周边站点贡献；徐汇区排名第二，但平均利用率更高，说明其站点供给更紧张。

### 关键指标

根据问题自动选择 2 到 4 个指标。

示例：

- 总收入：126.8 万元
- 环比变化：+8.6%
- 最高区域：浦东新区
- 平均利用率：31.2%

### 图表

图表类型根据问题类型选择：

| 问题类型 | 图表 |
|---|---|
| 趋势类 | 折线图 |
| 排名类 | 柱状图 |
| 对比类 | 分组柱状图 |
| 占比类 | 环形图 |
| 时段类 | 小时热力图 |
| 异常类 | 表格 + 地图高亮 |
| 预测类 | 折线图 + 预测虚线 |

图表可读性要求：

- 图表必须展示标题、关键轴标签或分类标签。
- 柱状图需要展示主要分类名称，不能只有无说明的柱子。
- 图表下方需要说明数据来源和统计口径。
- 当用户问题包含多个指标或多个意图时，回答不能只返回其中一个指标；需要拆分成多个结论或在同一张表中并列展示。

### 明细表格

表格默认展示聚合结果，不展示敏感明细。

默认限制：

- 最多展示 20 行
- 可显示“查看更多”但第一版可不做真实分页
- 金额、百分比、时间统一格式化

### 追问建议

每次回答后给 3 个追问建议。

示例：

- 只看浦东新区呢？
- 这些站点的故障情况如何？
- 按工作日和休息日拆开看

## 可信解释区

可信解释默认折叠，用户可展开。

包含：

- Schema 匹配结果
- 匹配理由
- 生成 SQL
- 查询结果摘要
- 安全校验说明

### Schema 展示示例

| 匹配项 | 结果 |
|---|---|
| 命中表 | fact_station_daily, dim_region |
| 命中字段 | total_revenue, region_name, stat_date |
| 指标口径 | 充电收入 = total_revenue 求和 |
| 匹配理由 | 问题包含“区域”和“收入排名” |
| 置信度 | 0.91 |

### SQL 展示策略

SQL 可以展示，但不应抢占主体验。

建议：

- 默认折叠
- 使用等宽字体
- 标注“只读查询”
- 对失败 SQL 显示修正后的版本

## 多轮对话策略

系统需要维护上下文：

- 上一轮时间范围
- 上一轮区域或站点
- 上一轮指标
- 上一轮结果集合
- 用户指代词，例如“这些站点”“它们”“这个区域”

### 追问类型

| 类型 | 示例 | 处理方式 |
|---|---|---|
| 缩小范围 | 只看浦东呢？ | 继承指标和时间，追加区域筛选 |
| 换指标 | 再看利用率 | 继承时间和区域，替换指标 |
| 追问原因 | 为什么下降？ | 基于上一轮对象做归因拆解 |
| 指代消解 | 这些站点故障多吗？ | 继承上一轮结果集合 |
| 图表延展 | 按周展示 | 调整聚合粒度 |

## 澄清交互

当问题存在歧义时，系统不直接猜测，而是澄清。

### 需要澄清的情况

- 指标口径不明确
- 时间范围不明确且可能影响结论
- 区域名称有歧义
- 用户提到“最好”“最差”但未说明指标

示例：

用户：

> 哪些站点表现最好？

系统：

> “表现最好”可以按收入、订单量、利用率或健康度判断。你想优先看哪个指标？

可选按钮：

- 收入最高
- 订单量最高
- 利用率最高
- 健康度最高

## 错误处理

### 超范围问题

用户问非充电运营相关内容时，系统解释范围并推荐可问问题。

示例：

> 当前 Demo 主要支持城市充电网络运营数据，包括收入、订单、利用率、故障、运维、天气和节假日影响。你可以试试问：“最近 30 天各区域收入排名如何？”

### 权限和隐私问题

用户查询手机号、车牌、个人明细时，系统拒绝并给出替代问题。

示例：

> 这个问题涉及个人敏感信息，当前 Demo 不支持查询手机号、车牌或可识别个人身份的数据。你可以改问“不同车辆类型的平均充电时长”或“各区域用户分群订单量”。

### SQL 失败

SQL 失败时，系统不暴露原始错误堆栈。

展示：

- 查询失败原因的业务化解释
- 系统尝试修正的方向
- 推荐相近问题

## 视觉规范

参考个人网站风格，采用浅色专业 SaaS。

### 色彩

| 用途 | 建议 |
|---|---|
| 页面背景 | 米白或浅灰白 |
| 主色 | 暖橙/棕红 |
| 正文 | 深黑或深灰 |
| 辅助色 | 灰蓝、蓝绿 |
| 风险色 | 柔和红 |
| 成功色 | 低饱和绿 |

### 组件风格

- 卡片圆角与个人网站保持一致
- 按钮使用暖橙主色
- 图表避免高饱和配色
- 地图区域低饱和、轻量化
- 技术信息放入折叠区，避免界面过重

## 首屏建议文案

标题：

> ChargeBI 智能充电运营问数助手

副标题：

> 在地图上看懂上海都市圈充电网络，并用自然语言追问收入、利用率、故障和运营机会。

问候语：

> 你好，我是 ChargeBI。你可以像问数据分析师一样问我：哪个区域收入最高、哪些站点利用率偏低、故障是否影响收入，或者未来 7 天哪里可能出现高峰压力。

## 验收标准

页面原型达到以下标准，才进入 PRD 和实现：

- 用户进入页面 5 秒内能理解场景
- 用户不用阅读长说明就知道可以提问
- 推荐问题能覆盖核心能力
- 地图和问答能互相影响上下文
- 每次查询都有结论、图表、表格和追问
- Schema 与 SQL 展示不干扰业务体验
- 错误处理能引导用户继续使用
