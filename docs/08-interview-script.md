# 面试讲解稿 v1.0

## 30 秒版本

ChargeBI 是我做的一个智能问数产品 Demo，场景是城市充电网络运营。它面向充电运营公司的业务人员，让他们不用写 SQL，也能直接用自然语言分析收入、订单、利用率、故障和运营机会。

这个项目我重点展示的不是一个聊天框，而是一套完整的 AI 问数链路：问题理解、Schema 匹配、SQL 生成、安全校验、查询执行、结果解释、图表展示和多轮追问。

## 2 分钟版本

我做 ChargeBI 的起点是传统 BI 的一个痛点：固定看板可以回答预设问题，但业务人员遇到临时问题时，还是要找研发或数据分析师写 SQL。

我先比较了智慧城市、工业供应链、美食出行、园区能碳等方向，最后选择城市充电网络运营。原因是这个场景既有地图展示，又有天然适合问数的运营指标，比如收入、订单、利用率、故障率、峰谷时段和区域排名。

产品第一屏采用左侧 AI 问数工作台、右侧地图运营视图。用户可以直接问“最近 30 天各区域收入排名如何”，系统会展示它如何理解问题、匹配哪些表和字段、生成什么 SQL，然后返回结论、图表、表格和地图高亮。

为了让 Demo 稳定，我没有完全信任模型输出，而是设计了产品化护栏：只读 SQL、白名单表、敏感信息拦截、超范围问题引导、歧义澄清和规则兜底。这样既能体现 AI 能力，也能体现真实产品需要的安全边界和可控性。

这个项目采用 SDD 方式推进，所有产品判断、数据设计、交互流程、技术方案和代码变化都沉淀在 Markdown 文档里，再进入实现。这也是我想展示的 AI 产品经理能力：不只是会提想法，也能把想法结构化并落成可体验的产品。

## 5 分钟结构

### 1. 为什么做

传统 BI 解决固定看板问题，但临时问数仍然依赖技术人员。AI 问数的机会在于，让业务人员直接从数据库中获得答案。

### 2. 为什么选充电运营

选择标准有三个：

- 数据是否可构建
- 用户是否一眼能理解
- 是否适合展示问数链路

充电运营满足这些条件，并且有智慧城市、新能源、地图和运营分析的组合优势。

### 3. 产品如何设计

产品不是单独的聊天框，而是 AI 数据分析工作台。

核心模块：

- 问数输入
- 推荐问题
- 多轮对话
- 工作流进度
- 结果图表
- 地图联动
- Schema 和 SQL 可信解释

### 4. 数据如何设计

数据库包含区域、充电站、充电桩、充电会话、站点日汇总、故障工单、运维记录、电价规则、日历和天气表。

这些表支撑趋势、排名、对比、异常、归因、预测和建议类问题。

### 5. AI 链路如何控制风险

开放式输入会带来不稳定，所以我设计了分层护栏：

- 问题超范围时不进入 SQL
- 涉及隐私时直接拒绝
- SQL 必须只读
- 只能访问白名单表
- 查询失败时给出修正建议
- 核心问题有规则兜底

### 6. 我在项目中体现的能力

- 产品选题判断
- AI 链路设计
- 数据库和指标建模
- 交互和错误状态设计
- SDD 文档化协作
- vibecoding 落地能力

## 高频追问回答

### Q1: 为什么不直接做通用 BI 问数？

通用问数范围太大，Demo 容易失控。我选择一个垂直场景，是为了让数据、指标、用户问题和交互都更真实，也更适合作品集展示。

### Q2: 模拟数据会不会影响可信度？

这个 Demo 的定位是展示产品能力，不是披露真实商业数据。数据策略是参考公开数据结构，再模拟运营数据，并在页面明确说明。重点是数据库关系、指标口径和问数链路是否合理。

### Q3: 为什么要展示 Schema 和 SQL？

因为智能问数产品最大的问题是可信度。业务人员不一定要看 SQL，但面试官和高级用户需要知道系统为什么这么查。默认折叠，既不打扰业务体验，也保留可信解释。

### Q4: 真实上线还需要补什么？

需要补真实数据接入、用户权限、数据治理、指标血缘、SQL 审计、模型评估集、生产监控和更强的安全策略。
