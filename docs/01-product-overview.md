# ChargeBI 产品方案总览 v1.0

## 产品名称

ChargeBI 智能充电运营问数助手

## 一句话定位

面向充电运营公司的 AI 问数 Copilot，让业务人员通过自然语言分析城市充电网络的收入、利用率、故障、区域表现和运营机会。

## 作品集定位

ChargeBI 是个人网站中的 AI 产品 Demo，用于展示 AI 产品经理从 0 到 1 完成智能问数产品设计与 vibecoding 落地的能力。

重点展示能力：

- 选择真实、有业务价值、适合 AI 化的场景
- 设计完整 AI 问数链路，而不是只做聊天框
- 构建数据、指标、Schema、交互和错误处理闭环
- 做出可体验、可演示、可解释的 Web Demo

## 场景选择

最终选择城市充电网络运营分析。

选择原因：

| 维度 | 判断 |
|---|---|
| 数据获取 | 有公开充电桩和充电会话数据可参考，可用模拟数据补全 |
| 数据库构建 | 适合建表：站点、充电桩、订单、故障、运维、区域、天气、节假日 |
| 用户理解 | 地图、充电站点、经营指标组合后，普通访客容易理解 |
| 问数适配 | 收入、订单、利用率、故障率、峰谷时段、区域排名都适合自然语言查询 |
| 展示效果 | 能体现智慧城市、新能源、运营分析、AI Copilot |
| 创意感 | 比普通客服或报表 Demo 更有空间感和行业质感 |

## 数据策略

采用“上海都市圈示例数据 = 公开数据参考 + 模拟运营数据构建”。

页面可说明：

> 本 Demo 基于公开数据结构与模拟运营数据构建，用于展示 AI 问数产品的核心体验，不代表真实商业运营数据。

数据范围：

- 时间范围：最近 12 个月
- 数据粒度：每笔充电会话
- 空间范围：上海核心区域与周边示例区域

建议区域：

- 浦东新区
- 徐汇区
- 静安区
- 黄浦区
- 闵行区
- 嘉定区
- 松江区
- 临港新片区
- 虹桥枢纽
- 张江
- 陆家嘴
- 人民广场

## 目标用户

第一版只服务一个核心角色：运营分析经理。

目标用户的典型问题：

- 哪些区域收入最好？
- 哪些站点利用率低？
- 最近订单为什么下降？
- 故障是否影响收入？
- 节假日和工作日的充电行为有什么差异？
- 下周哪些站点可能出现高峰压力？
- 应该优先优化哪些站点？

## 核心价值

传统 BI 看板的问题：

- 只能看固定指标
- 临时问题需要找研发或数据分析师
- 不知道指标在哪张表、哪个字段
- 看到了数字，但不知道原因
- 不支持连续追问

ChargeBI 的价值：

- 用户直接用自然语言问数
- 系统自动理解问题、匹配 Schema、生成 SQL、校验并查询
- 返回业务结论、图表、表格和追问建议
- 支持地图联动和多轮上下文
- 出错时主动解释并引导用户修正

## 核心体验结构

首页第一屏采用左右布局：

- 左侧：AI 问数工作台
- 右侧：地图和运营指标总览

左侧负责：

- 自然语言输入
- 推荐问题
- 多轮对话
- AI 分析结论
- 图表
- 明细表格
- 可信解释

右侧负责：

- 上海都市圈示例充电网络地图
- 站点点位
- 区域聚合
- 站点状态
- 关键运营指标

产品形态不是传统大屏，而是带地图能力的 AI 数据分析工作台。

## 视觉风格

ChargeBI 应与个人网站风格保持一致。

视觉方向：

- 浅色专业 SaaS
- 暖橙科技点缀
- 米白背景
- 深黑标题
- 浅灰卡片
- 地图区域使用低饱和色
- 图表使用橙色、蓝绿色、灰蓝色辅助

避免：

- 过度深色科技大屏
- 高饱和霓虹风格
- 与个人网站割裂的企业后台感

## MVP 功能范围

第一版必须包含：

- 自然语言问数输入
- 推荐问题引导
- 多轮上下文对话
- 问题理解展示
- AI 工作流进度展示
- Schema 匹配展示
- SQL 生成与折叠展示
- SQL 校验
- 数据库查询
- AI 总结结论
- 图表生成
- 明细表格
- 地图联动
- 追问建议
- 超范围问题引导
- 歧义问题澄清
- 权限和隐私类错误处理
- 简单预测与运营建议

第一版不做：

- 用户登录
- 多角色权限系统
- 真实支付或订单接入
- 实时地图定位
- 复杂机器学习预测
- 生产级数据权限系统
- 多租户 SaaS 后台

## 支持问题类型

| 类型 | 示例 |
|---|---|
| 趋势类 | 最近 30 天充电收入趋势怎么样？ |
| 排名类 | 哪些区域订单量最高？ |
| 对比类 | 浦东和徐汇哪个区域利用率更高？ |
| 异常类 | 上周哪些站点故障率异常？ |
| 归因类 | 为什么本月订单量下降？ |
| 预测类 | 未来 7 天哪些站点可能出现高峰？ |
| 建议类 | 哪些低利用率站点需要优化？ |
| 追问类 | 那这些站点主要是什么故障？ |

## AI 链路

用户提问后，系统展示流程：

1. 正在理解问题
2. 正在识别时间、区域、指标和分析意图
3. 正在匹配数据表与字段
4. 正在生成 SQL 查询
5. 正在校验 SQL 安全性
6. 正在执行查询
7. 正在生成分析结论与图表

可信解释区展示：

- 我理解你的问题
- 命中的指标
- 命中的表和字段
- 匹配理由
- 生成 SQL
- 查询结果摘要

## 技术策略

- 模型：Kimi 或 DeepSeek
- 数据库：优先 SQLite 或 DuckDB，保证部署简单稳定
- 前端：根据最终 PRD 决定，倾向 React 或 Next.js
- 地图：前期可用 Leaflet、Mapbox 或高德地图 Web JS API
- 部署：先做独立 Web App，再从个人网站作品详情页进入或嵌入

## 作品集呈现方式

产品 Demo 和作品集说明分离。

个人网站上建议做两层：

1. 作品集详情页：解释需求背景、用户痛点、AI 链路、数据构建、关键交互、vibecoding 过程和最终效果。
2. 独立 Demo 入口：进入后就是完整 ChargeBI 产品界面，不混入项目说明。
