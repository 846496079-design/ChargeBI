# 决策日志

本文件记录 ChargeBI 项目中的关键决策、想法变化、策略变化和范围调整。

## 2026-06-20

### D001: 产品目标从真实企业落地转为作品集 Demo

- 决策：当前目标不是服务某个真实企业场景，而是做一个可上个人网站的智能问数作品集 Demo。
- 原因：用户希望展示 AI 产品经理能力和 vibecoding 能力。
- 影响：需要优先保证可理解、可演示、可部署、可解释。

### D002: 选题从泛智慧城市收敛到城市充电网络运营

- 决策：选择新能源充电桩运营方向。
- 原因：兼具智慧城市、新能源、运营分析、AI Copilot；适合地图展示和问数。
- 备选：工业园区能碳、美食/出行规划、共享单车、城市应急。
- 影响：数据库围绕站点、充电桩、会话、收入、故障、运维、区域、天气、节假日设计。

### D003: 美食/出行作为交互灵感，不作为主选题

- 决策：不把美食或出行规划作为主产品方向。
- 原因：其核心更偏搜索、推荐和路线规划，不如充电桩适合展示 Schema 匹配、SQL 生成和智能问数。
- 影响：可以借鉴地图入口和低门槛引导，但产品核心仍是运营问数。

### D004: 数据采用“上海都市圈示例数据 + 公开数据参考 + 模拟补全”

- 决策：产品体验包装为上海都市圈，但不承诺全部来自上海真实明细数据。
- 原因：上海有真实充电基础设施平台和行业体感，但公开可下载的完整运营明细不足。
- 影响：页面需要说明数据为公开数据结构与模拟运营数据构建，用于作品集 Demo。

### D005: 产品名确定为 ChargeBI 智能充电运营问数助手

- 决策：产品名称为 ChargeBI 智能充电运营问数助手。
- 原因：既体现充电场景，也体现 BI 问数属性。
- 影响：后续文档、页面和 Demo 统一使用该名称。

### D006: 首页形态确定为左侧 AI 工作台 + 右侧地图指标

- 决策：第一屏采用左侧 AI 问数工作台，右侧地图和运营指标。
- 原因：既能体现 AI Copilot，又能让用户通过地图快速理解场景。
- 影响：页面原型围绕双栏工作台设计。

### D007: AI 策略采用真实 LLM + 产品化护栏

- 决策：支持开放式自然语言输入，使用 Kimi 或 DeepSeek，并通过白名单、SQL 校验、查询模板兜底保证稳定。
- 原因：作品集需要真实体验，但 Demo 必须稳定可控。
- 影响：后续技术方案必须包含 SQL 安全、Schema 限制、错误恢复和模板兜底。

### D008: 可视化包含地图，但地图可按体验质量调整

- 决策：前期尝试地图；如果体验不佳，可以降低地图复杂度或拿掉部分交互。
- 原因：地图能增强第一眼理解，但不应成为工程风险。
- 影响：地图先做站点点位、区域聚合和选区联动，不做复杂实时地理能力。

### D009: 预测采用简单趋势外推 + 规则建议

- 决策：第一版不做复杂机器学习预测。
- 原因：预测不是核心卖点，过度投入会拉高工程风险。
- 影响：预测问题由近 30 天或近 90 天趋势外推和运营规则解释支撑。

### D010: Demo 产品和作品集讲解分离

- 决策：产品界面只呈现 ChargeBI 产品本身；作品集详情页单独讲设计过程。
- 原因：用户明确要求展示产品时应完全是产品本身，不能和说明文档混为一谈。
- 影响：后续需要分别设计 Demo Web App 和作品集详情页。

### D011: 视觉风格需贴合个人网站

- 决策：采用浅色专业 SaaS + 暖橙科技点缀。
- 原因：用户个人网站已有浅色、暖橙、卡片式、产品展示风格。
- 影响：避免深色大屏和过度科技风。

### D012: 所有文件使用 Markdown，并遵循 SDD

- 决策：所有规划类产物统一为 `.md` 文件；所有想法变化、策略变化和后续代码修改都要落实到文档。
- 原因：用户明确要求。
- 影响：新增 `docs/00-sdd-index.md` 作为文档入口，后续实现前需同步规格文档。

### D013: 页面原型采用产品体验优先，而非作品集说明混排

- 决策：ChargeBI Demo 首页只呈现产品功能，不在产品界面中解释项目背景、设计过程或作品集叙事。
- 原因：用户明确要求“展示产品的时候就完全是产品本身”。
- 影响：Demo 页面采用独立产品工作台；作品集详情页后续单独设计。

### D014: 初始页面需要强引导，以降低陌生用户理解成本

- 决策：在产品第一屏保留推荐问题、示例提示、地图默认选区和运营指标总览。
- 原因：开放式输入虽然自由，但陌生访客需要一眼知道能问什么。
- 影响：页面原型中必须包含推荐问题区、空状态引导和超范围问题推荐。

### D015: PRD 将 Demo 定义为作品集可展示完成度，而非生产级 SaaS

- 决策：ChargeBI v1.0 的完成度目标为作品集可展示，体验完整但不追求生产级。
- 原因：用户选择 B 级完成度，希望可上个人网站展示，同时控制开发成本。
- 影响：PRD 中保留登录、多租户、生产级权限、复杂预测等为非本期范围。

### D016: 技术方案优先选择 Next.js + SQLite 的轻量全栈路径

- 决策：技术方案建议优先采用 Next.js 全栈应用，SQLite 作为本地文件数据库。
- 原因：用户要求部署简单稳定，个人网站技术栈未定死；Next.js 便于同时承载前端页面、API、部署和作品集入口。
- 影响：后续实现默认以 Next.js 为主，除非实际部署环境要求调整。

### D017: AI 问数链路采用分层护栏，而不是完全信任模型输出

- 决策：LLM 负责理解、Schema 匹配、SQL 生成和总结，但 SQL 执行前必须经过白名单、只读、表字段、LIMIT 和敏感信息校验。
- 原因：开放式输入必须兼顾真实体验和 Demo 稳定性。
- 影响：技术实现中必须包含 Query Guard 模块。

### D018: 初版先提供规则兜底问数链路，预留真实模型接入

- 决策：初版 Web Demo 优先实现稳定可体验的规则兜底链路，同时保留 DeepSeek/Kimi 的接入结构。
- 原因：用户要求一直推进到初版完成；在没有 API Key 和部署环境确认前，稳定 Demo 比阻塞在模型接入更重要。
- 影响：`/api/query` 需要支持核心样例问题的意图识别、SQL 生成、图表和解释；后续可替换为真实 LLM。

### D019: 地图从示意散点升级为真实底图

- 决策：不再继续优化 CSS 示意地图，改用 Leaflet 真实地图底图展示上海区域与站点经纬度。
- 原因：用户反馈当前地图表达效果仍然较差；示意地图缺少真实地理认知锚点，难以体现“城市充电网络”。
- 影响：前端需要引入 Leaflet，地图组件改为真实经纬度点位、状态图例、站点弹窗和结果高亮。

### D020: 智能问数不能以规则路由作为核心能力

- 决策：规则路由只能作为 Demo 稳定兜底，不应作为智能问数主链路。
- 原因：用户自然语言 query 千奇百怪，可能包含多指标、多时间、多维度、比较、排序、归因和条件组合；靠人工枚举规则无法覆盖真实问数需求。
- 影响：后续架构需要升级为 LLM-first 的语义解析、查询规划、Schema Linking、SQL 生成、校验执行和结果反思链路；规则模板只保留为推荐问题和失败兜底。

### D021: 问数链路必须先判断权限和数据可得性

- 决策：用户 query 进入后，必须先判断是否有权限查询、数据库是否有相关数据、现有数据是否能计算出目标指标。
- 原因：如果数据库没有相关字段或无法推导指标，系统不能乱查、乱生成或编造数据，否则会造成幻觉。
- 影响：LLM-first 查询链路中需要新增 Data Availability Check；缺少数据时必须明确告知缺少哪些数据，以及可替代查询方向。

### D022: 地图与问数结果需要双向联动

- 决策：当问数结果提到站点时，地图应高亮这些站点并淡显其他站点；当用户点击结果表格或图表中的某个站点时，地图应单点高亮。
- 原因：地图不只是展示背景，而应成为查询结果的空间解释层。
- 影响：前端需要保存 focusedStationId 和 queryHighlightStationIds，并传入地图组件控制 marker 样式。

### D023: 多意图 query 必须拆成多个子查询分别回答

- 决策：当用户在一个 query 中同时询问多个独立指标或排序目标时，系统应拆成多个子查询，而不是合并或丢弃其中一部分。
- 原因：例如“亏损额最高是谁？营业额最高是谁？净利率最高是谁？”包含 3 个并列意图，不能理解成“亏损额最高者的净利率是多少”。
- 影响：查询计划层需要支持 `sub_queries`；当前实现先为站点经营极值类问题增加专门多子查询处理。

### D024: 区域范围从近似矩形升级为真实 GeoJSON 边界

- 决策：地图区域按钮不再使用手写 bounds 矩形框，改为使用上海区县 GeoJSON 行政边界。
- 原因：用户希望区域框选更真实，近似矩形只能表达大概范围，作品集质感不足。
- 影响：项目需要本地保存 GeoJSON 文件，Leaflet 根据区域名称匹配 polygon，并在点击区域按钮时 `fitBounds` 到真实边界。

### D025: 地图快捷区域统一为行政区

- 决策：地图顶部快捷区域统一使用行政区，不再混入虹桥枢纽、人民广场、陆家嘴、张江等业务区。
- 原因：业务区与行政区空间上会重叠，容易造成用户对区域边界的误解。
- 影响：快捷按钮只展示行政区；业务区仍可作为站点标签或查询维度，但不作为行政边界按钮。

### D026: 选中行政区后区外站点淡显

- 决策：用户点击行政区按钮后，地图高亮该行政区内站点，区外站点淡显。
- 原因：区域选择不仅要缩放和画边界，还要强化站点归属和空间过滤感。
- 影响：地图 marker 样式需要根据 selectedRegionName 判断是否 dimmed。

### D027: 问数主链路升级为 AI-first

- 决策：DeepSeek 不再只做意图识别，而是参与 QueryPlan 生成、Schema Linking、SQL 生成、结果解释和多意图子查询编排。
- 原因：如果 SQL 和回答都由规则模板完成，Demo 会显得“假智能”，无法体现真正智能问数能力。
- 影响：`/api/query` 主链路改为 AI-first；后端负责安全校验、执行和兜底。规则路由只在模型失败、SQL 不安全、无法解析时启用。

### D028: 用户反馈必须按问题类别系统修复

- 决策：用户指出的一个具体 query、页面异常或交互问题，默认视为一类问题的代表，不再只针对当前句子做局部 patch。
- 原因：智能问数产品的关键链路是“意图识别 -> 复杂任务编排 -> 结果表格 -> 可视化 -> 多轮上下文”，任一局部修复都可能影响整条链路。只修当前样例会导致拆东墙补西墙。
- 影响：后续每次修改必须至少回答四件事：问题类别是什么、根因在哪一层、系统性修复方案是什么、需要覆盖哪些回归用例。文档和代码必须同步体现这个分析。

### D029: 引入答案审查子 Agent 与指标量纲校验

- 决策：问数链路在生成答案后，新增独立的 Answer Verifier。Verifier 只读取用户原始 query、系统理解、SQL 摘要、表格/图表/KPI/最终结论，判断答案是否覆盖问题、指标是否绑定正确、占比分子分母是否同量纲。
- 原因：用户指出“亏损额 Top10 站点营业总额是多少，他们占总亏损额多少”被错误解释为“营业总额 / 总亏损额”，产生 2774% 这种跨指标占比幻觉。此类问题根因不是单句规则，而是 Planner、SQL、Summary 之间缺少执行后审查。
- 影响：后续 AI-first 链路必须包含两层审查：确定性 Metric Binding Guard 先拦截明显量纲错误；LLM Verifier 再审查语义覆盖、多意图遗漏和解释跑偏。审查失败时最多重试一次，仍失败则澄清或降级到稳定合同，避免死循环。

### D030: 指代消解与澄清反问必须形成连续任务

- 决策：句内已经出现明确对象时，例如“营业额最高的 10 个站点”“亏损额 Top10 站点”，后续“他们/这些站点”必须优先解析为句内对象，而不是要求外部上下文。
- 决策：当系统确实需要澄清时，澄清不是打断任务，而是保存 pending question；用户下一轮补充说明后，系统要把原问题和澄清内容合并，继续执行原任务。
- 原因：用户反馈只要出现“他们”就被拦截，且用户澄清后仍无法继续查询，导致体验像中断而不是对话式问数。
- 影响：后端需要扩展句内先行词识别，并在 clarification 的 `nextContext` 中保存 `pendingClarification`；下一轮优先尝试用澄清文本合并原始问题。

### D031: 规则必须降级为护栏，不再承担开放式业务理解

- 决策：ChargeBI 的规则体系分为三类：必须保留的护栏规则、可接受的确定性执行规则、需要迁出主链路的业务路由规则。
- 决策：安全、权限、数据可得性、SQL 只读校验、字段白名单、量纲一致性、答案审查属于护栏规则，应保留；自然语言意图识别、业务问题拆解、Schema Linking、SQL 生成、图表选择、结果解释应优先由 Agent Planner/Verifier 完成。
- 决策：临时稳定合同只允许作为 Demo 兜底和回归保护，不能继续无限扩张；每个稳定合同必须有对应的泛化目标和退出方向。
- 原因：用户 query 不能穷举，继续堆规则会造成误触发、异常输出和“假智能”体验。
- 影响：后续改造方向是收敛 `answerQuestionByRules` 和各类 `answerXXX` 业务合同，把规则从“路由答案”改为“审查候选答案和保护系统边界”。

### D032: 澄清必须是任务内续流，而不是重新提问

- 决策：澄清反问在交互上独立成卡片内输入框，用户在卡片内补充说明时，前端必须带上 `clarificationFor`，后端将补充说明合并回原问题继续执行。
- 决策：后端默认先走 Agent Planner，不再让“他们/这些”等指代规则抢在模型之前中断任务；指代规则只在 Planner 无法给出可靠答案后作为澄清护栏。
- 原因：用户反馈当前澄清体验像打断业务流程，且规则看到“他们”就中断，无法灵活理解句内上下文。
- 影响：`/api/query` 的主链路改为 Planner-first；稳定业务合同从默认主链路移出，只作为显式兜底或回归保护。

### D033: 多意图结果必须按子任务分块呈现

- 决策：复杂 query 不能只返回一张表、一个图和一段总摘要；每个子意图都应形成独立结果块，包含子结论、KPI、表格、图表和导出能力。
- 原因：用户期望智能问数能根据 query 灵活编排结果。如果用户问 3 个问题，系统需要召回 3 组数据、生成 3 组结论和 3 个图表，而不是压缩成一个结果。
- 影响：`AnswerBlock` 扩展 `summary` 和 `kpis`，前端按 block 渲染；Excel 导出也按 block 支持单组导出和全部导出。
