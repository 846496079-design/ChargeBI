# 实现记录

本文件记录 ChargeBI 代码实现过程中的重要修改、策略变化和与规格文档的对应关系。

## 2026-06-20

### I001: 建立 SDD 文档体系

- 修改：新增 README 和 `docs/` 下的 SDD 文档。
- 对应规格：`docs/00-sdd-index.md`
- 说明：确立所有想法变化、策略变化和代码修改都同步到 Markdown。

### I002: 确立初版实现策略

- 修改：决定初版先实现规则兜底问数链路，预留 DeepSeek/Kimi 接入。
- 对应规格：`docs/06-technical-implementation.md`
- 原因：没有 API Key 和部署环境确认时，稳定可体验的作品集 Demo 优先。

### I003: 准备进入 Web Demo 实现

- 计划修改：初始化 Next.js、SQLite、地图、图表、问答 API。
- 对应规格：`docs/05-prd.md`、`docs/06-technical-implementation.md`
- 验收：本地可运行，推荐问题可返回结论、图表、表格、地图高亮和可信解释。

### I004: 初始化 Next.js 项目和 SQLite 数据脚本

- 修改：新增 `package.json`、Next.js 基础配置、全局样式、SQLite 数据生成脚本和只读数据库访问层。
- 对应规格：`docs/06-technical-implementation.md`
- 说明：使用 Node 24 内置 `node:sqlite`，避免额外原生 SQLite 依赖，提升部署稳定性。

### I005: 修正站点表数据写入占位符

- 修改：`dim_station` 表有 17 个字段，修正 `INSERT` 语句占位符数量。
- 对应规格：`docs/03-data-design-and-sample-questions.md`
- 说明：这是数据生成脚本实现错误，不改变数据模型。

### I006: 调整初版样例数据生成规模和性能策略

- 修改：索引改为数据写入后创建，开启 SQLite 批量写入优化；站点和会话规模调整为初版演示足够的轻量中等版。
- 对应规格：`docs/06-technical-implementation.md`
- 原因：首次按较大规模生成超过两分钟，影响初版推进效率。
- 影响：初版仍保留 12 个月、10 张表、区域/站点/故障/天气/日历等完整结构，后续可放大数据量。

### I007: 修正 Next 构建兼容问题

- 修改：移除不必要的 Next experimental 配置，收窄 SQLite 查询参数类型。
- 对应规格：`docs/06-technical-implementation.md`
- 说明：保持项目使用标准 Next.js 配置，降低部署风险。

### I008: 实现初版问数 API 和前端工作台

- 修改：新增 `/api/overview`、`/api/stations`、`/api/query`，实现 ChargeBI 首页、问数工作台、地图点位、图表、表格、可信解释区。
- 对应规格：`docs/04-prototype-and-interaction.md`、`docs/05-prd.md`
- 说明：初版采用规则兜底 NL2SQL，支持区域收入排名、低利用率建议、故障异常、预测、归因、区域对比、站点贡献、隐私拦截和超范围引导。

### I011: 修正地图被对话撑高的问题

- 影响：页面整体更接近固定工作台，查询结果增多不会继续拉高地图。

### I012: 地图升级为 Leaflet 真实底图

- 影响：地图体验依赖在线瓦片服务；后续部署时可替换为高德或 Mapbox 底图。

### I013: 增强地图点位识别与详情卡关闭逻辑

- 影响：地图更像可交互运营地图，站点详情不再常驻遮挡底图。

### I014: 优化站点数量 Badge 排版并配置 DeepSeek 环境变量

### I017: SQL 护栏支持只读 CTE 查询

### I018: 增加初始化数据加载状态和错误提示

### I019: 修正亏损额和净利率复合意图

- 影响：系统现在会按站点维度同时返回示例亏损额最高站点和示例净利率最低站点，并解释示例财务口径。

### I020: 调整示例成本口径以支撑亏损分析

- 修改：将运维成本估算从收入 3.5% + 每单 0.8 元调整为收入 16% + 每单 0.8 元。

- 原因：原口径过于保守，样例数据中所有站点示例亏损额都为 0，不利于展示“亏损额最高”问题。
- 影响：亏损额和净利率问题现在具备更明显的站点差异。

### I021: 明确规则路由不是智能问数主架构

- 影响：下一阶段应将 `/api/query` 重构为 LLM-first 查询规划链路。

### I029: 修复行政区缩放与边界图层回归

- 修改：行政区选中优先级高于历史查询高亮；点击行政区会清空单站点聚焦。
- 对应规格：`docs/04-prototype-and-interaction.md`
- 原因：用户反馈选中行政区域时缩放逻辑和边界框消失。

### I030: 区内站点判断从名称匹配改为空间包含

- 修改：选中行政区后，站点是否高亮不再只看 `region_name`，而是优先判断站点经纬度是否落在 GeoJSON polygon 内。
- 对应规格：`docs/04-prototype-and-interaction.md`
- 原因：模拟数据中部分站点名称属于某行政区，但坐标可能落在真实行政边界外；名称匹配会导致区外点未淡显。
- 影响：地图视觉以真实空间位置为准；后续可进一步修正模拟数据生成逻辑，让站点坐标与行政区标签一致。

### I031: 站点数据生成约束到行政区 GeoJSON 内

- 修改：数据生成脚本加载 WGS84 上海区县 GeoJSON，并在生成行政区站点时采样真实 polygon 内点位。
- 修改：人民广场归属黄浦区，陆家嘴和张江归属浦东新区，虹桥枢纽归属闵行区，确保业务标签站点坐标仍落在对应行政区内。
- 对应规格：`docs/03-data-design-and-sample-questions.md`、`docs/04-prototype-and-interaction.md`
- 原因：用户要求从数据源头修复站点名称、region_id 和坐标不一致问题。

### I032: 增加全区域按钮与三意图经营问题

- 修改：地图区域快捷按钮新增“全区域”，用于退出行政区筛选并恢复全局视角。
- 修改：新增“亏损最严重 / 盈利最多 / 利用率最低”三意图站点问题处理，拆成 3 个子查询分别回答。
- 对应规格：`docs/04-prototype-and-interaction.md`、`docs/06-technical-implementation.md`
- 原因：用户反馈复杂 query 仍被压缩成单一意图；区域筛选后缺少返回全局视图的入口。

### I033: 修正地图缩放控件与营收极值对比问题

- 修改：回答卡片展示 DeepSeek 是否参与意图识别，避免 AI 调用状态不透明。
- 对应规格：`docs/04-prototype-and-interaction.md`、`docs/06-technical-implementation.md`

### I034: 暴露 DeepSeek 调用状态 Trace

- 修改：`/api/query` 返回 DeepSeek 配置、调用、成功、状态码、耗时和错误类型，不返回密钥。
- 修改：前端问数链路区域展示模型调用状态，模型不可用时明确说明使用规则兜底。
- 对应规格：`docs/06-technical-implementation.md`

### I035: 问数链路升级为 AI-first

- 修改：新增 DeepSeek QueryPlan 生成、AI SQL 生成、AI 结果解释链路。
- 修改：规则路由降级为兜底，仅在模型失败、SQL 未通过安全校验或计划不可执行时使用。
- 对应规格：`docs/02-decision-log.md`、`docs/06-technical-implementation.md`
- 原因：用户要求 QueryPlan、Schema Linking、SQL 生成、结果解释和多意图编排都用上 AI，避免 Demo 显得像规则模板。

### I036: 增加结果后处理和比较型可视化编排

- 修改：AI 查询结果进入前端前增加结果后处理，支持最高 vs 最低场景自动生成差值行。
- 修改：新增 comparison chart，用于展示营业额、订单量、利用率、故障率等多指标对比。
- 修改：结果后处理增加中英文字段别名归一，避免 AI SQL 返回 `revenue`、`orders`、`utilization` 等字段时无法生成差值行。
- 修改：comparison chart 改为按指标独立归一缩放，避免营业额和利用率等不同量纲共用全局刻度导致可视化失真。
- 对应规格：`docs/06-technical-implementation.md`
- 原因：用户反馈表格缺少“差值”行，图表无法体现 query 真正关注的比较结果。

### I037: 增加多图形智能编排与标签避让

- 修改：QueryPlan 的 `chartType` 扩展为 `bar`、`line`、`pie`、`bubble`、`table`、`forecast`。
- 修改：后端增加选图器，按 query 语义和数据形态选择图形：占比/构成优先饼图，趋势/按天优先折线图，影响/权重/关系优先气泡分布图，排名/Top 优先柱状图。
- 修改：前端新增 `pie` 和 `bubble` 渲染，并保留查询结果表作为数据底座。
- 修改：柱状图横轴标签在数量多或文本长时自动倾斜展示，增加底部空间，减少重叠遮挡。
- 对应规格：`docs/06-technical-implementation.md`
- 原因：用户反馈图形不能只是一种形态，且当前柱状图站点名称发生文字重叠。

### I038: 结果分块、字段中文化与表图一致

- 修改：AI 多子查询结果改为 `blocks` 分块展示，每个子查询独立形成一张表和一张图，不再强行拼成稀疏大表。
- 修改：默认先展示主要结果；确有最高 vs 最低等比较语义时，才额外合成对比表和对比图。
- 修改：后端对 SQL 字段做中文化映射，例如 `top30_revenue` -> `前30站点营业额`，`avg_utilization` -> `平均利用率`。
- 修改：单行多指标结果不再默认画气泡图；收入类问题只抽取收入相关字段作图，其余指标保留在表格中。
- 对应规格：`docs/06-technical-implementation.md`
- 原因：用户反馈图和表没有对应关系、英文技术字段外露，并且多个查询结果不应强行塞进一张图或一张表。

### I039: 增加复杂 Query 意图合同校验

- 修改：新增“营业额 TopN + 亏损额 TopN + 各自整体占比”复杂 query 的意图合同编排。
- 修改：该类 query 固定拆成 2 个子查询：营业额 TopN 及整体营业额占比、示例亏损额 TopN 及整体亏损额占比。
- 修改：fallback 不再使用模型 `normalizedQuestion` 覆盖用户原始问题，避免误识别后把 query 改写成另一个问题。

### I048: 排查规则过重导致异常输出的问题

- 排查结论：当前主链路已经是 AI-first，但仍存在较多业务路由规则和稳定合同，尤其集中在 `lib/query/orchestrator.ts` 的 `answerQuestionByRules` 与多个 `answerXXX` 合同函数。
- 风险分类：安全、权限、数据可得性、SQL Guard、量纲一致性和答案审查属于必要护栏；关键词意图路由、固定 SQL、固定图表和固定总结属于需要逐步迁出的业务规则。
- 具体风险：规则在开放式 query 中容易误触发，例如把“亏损的主因”和“营业额最高”拼成“亏损额最高”，或把带有“他们”的比较问题错误降级为其他默认查询。
- 策略变化：后续不再继续无限增加业务合同；新增能力优先改 Planner、QueryPlan schema、Verifier 和澄清机制。稳定合同仅保留为 Demo 关键路径兜底，并必须记录退出方向。
- 对应规格：`docs/02-decision-log.md`、`docs/06-technical-implementation.md`
- 原因：用户指出 Agent 产品中除固定业务流程外应少用规则，用户 query 无法穷举，规则过重会导致异常输出。
