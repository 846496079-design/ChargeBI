# ChargeBI 产品需求文档 v1.0

## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 产品名称 | ChargeBI 智能充电运营问数助手 |
| 文档版本 | v1.0 |
| 文档类型 | PRD |
| 开发方式 | SDD，规格驱动开发 |
| 目标形态 | 可运行、可上个人网站的作品集 Demo |
| 目标用户 | 充电运营公司的运营分析经理 |

## 2. 背景

传统 BI 看板适合查看固定指标，但不适合回答临时业务问题。非技术业务人员在面对新问题时，通常需要找研发或数据分析师协助取数、写 SQL、解释结果。

ChargeBI 希望通过 AI 问数能力，让业务人员直接用自然语言查询城市充电网络运营数据，并获得结论、图表、表格和追问建议。

本产品作为个人网站作品集 Demo，用于展示 AI 产品经理在场景选择、数据建模、AI 工作流设计、交互设计、错误处理和 vibecoding 落地方面的能力。

## 3. 产品目标

### 3.1 用户目标

让运营分析经理无需写 SQL，即可回答以下问题：

- 哪些区域收入最高？
- 哪些站点利用率偏低？
- 哪些站点故障率异常？
- 节假日与工作日充电行为有什么差异？
- 本月订单为什么下降？
- 未来 7 天哪些站点可能出现高峰压力？
- 哪些站点应该优先运维或优化？

### 3.2 作品集目标

让招聘方或面试官看到：

- 这是一个真实可理解的 AI 产品场景
- 产品不是简单聊天框，而是完整智能问数链路
- Demo 具备数据、Schema、SQL、图表、地图和多轮对话闭环
- 产品设计过程与最终产品体验能够分离呈现

### 3.3 技术目标

- 支持开放式自然语言输入
- 支持 Kimi 或 DeepSeek 等国内大模型 API
- 使用 SQLite 或 DuckDB 保证部署简单稳定
- 通过 SQL 白名单、Schema 限制和模板兜底提升稳定性
- 可部署为独立 Web App，并从个人网站跳转或嵌入

## 4. 非目标

v1.0 不做：

- 用户登录
- 多角色权限系统
- 多租户 SaaS 后台
- 真实商业数据接入
- 实时支付或实时充电站状态接入
- 生产级数据治理平台
- 复杂机器学习预测模型
- 真实个人用户、手机号、车牌等敏感数据查询

## 5. 用户画像

职责：

## 6. 核心场景

### 6.1 场景一：区域经营分析

用户打开 ChargeBI，想了解最近 30 天哪个区域收入最高。

系统需要：

- 理解时间范围为最近 30 天
- 识别指标为充电收入
- 识别维度为区域
- 匹配 `fact_station_daily` 和 `dim_region`
- 生成区域收入排名
- 输出结论、柱状图、表格和地图高亮
- 给出追问建议

### 6.2 场景二：站点低利用率优化

用户问“哪些站点利用率最低，可能需要优化？”

系统需要：

- 查询低利用率站点
- 排除维护中和严重故障站点，避免误判
- 展示地图高亮
- 结合站点类型、时段订单和区域标签给出建议

### 6.3 场景三：故障异常排查

用户问“上周故障率最高的 5 个站点是哪些？”

系统需要：

- 识别上周时间范围
- 查询故障工单和站点桩数
- 计算故障率
- 展示站点列表、故障类型分布和地图高亮
- 支持继续追问“这些站点收入损失多少？”

### 6.4 场景四：多轮归因分析

用户问“为什么本月订单量下降？”

系统需要：

- 对比本月与上月订单量
- 拆解区域、站点、时段、天气、故障因素
- 给出主要原因排序
- 支持追问“主要看徐汇”或“是不是故障变多？”

### 6.5 场景五：预测与建议

用户问“未来 7 天哪些站点可能出现高峰压力？”

系统需要：

- 根据近 30 天或近 8 周趋势外推
- 结合工作日、休息日、节假日和时段模式
- 输出高风险站点列表
- 给出提前巡检、临时调度或扩容建议

## 8. 功能需求

### F001 自然语言问数输入

优先级：P0

需求：

- 用户可以输入开放式自然语言问题
- 支持回车提交
- 支持点击推荐问题提交
- 加载中防止重复提交
- 输入为空时不可提交

验收标准：

- 用户可以完成一次从输入到结果展示的完整查询
- 非数据库相关问题会进入超范围引导

### F002 推荐问题引导

优先级：P0

需求：

- 首页展示 4 到 6 个推荐问题
- 查询完成后展示 3 个上下文追问建议
- 查询失败后展示 2 到 3 个可问方向

验收标准：

- 新用户进入页面后无需阅读说明，即可点击推荐问题体验核心能力

### F003 AI 工作流进度展示

优先级：P0

需求：

用户提交后展示以下步骤：

1. 理解问题
2. 识别指标、时间和区域
3. 匹配数据表和字段
4. 生成 SQL
5. 校验查询安全性
6. 执行查询
7. 生成分析结论

验收标准：

- 查询时用户能看到 AI 正在做什么
- 查询结束后保留可展开的执行过程摘要

### F004 问题理解展示

优先级：P0

需求：

- 系统需要展示“我理解你的问题是……”
- 同时展示结构化理解结果，包括意图、时间、指标、维度和筛选条件

验收标准：

- 对推荐问题和常见自由输入问题，系统能输出合理理解

### F005 Schema 匹配

优先级：P0

需求：

- 系统根据用户问题匹配相关表和字段
- 展示命中的表、字段、指标口径、匹配理由和置信度
- Schema 展示默认折叠

验收标准：

- 区域收入、站点利用率、故障异常等问题能匹配到正确核心表

### F006 SQL 生成与安全校验

优先级：P0

需求：

- 只允许只读 SELECT 查询
- 查询表必须在白名单内
- 禁止 `DELETE`、`UPDATE`、`INSERT`、`DROP`、`ALTER`、`CREATE`
- 默认添加结果行数限制
- SQL 默认折叠展示

验收标准：

- 系统不会执行写操作
- 超出白名单的查询被拦截

### F007 查询执行

优先级：P0

需求：

- 根据 SQL 查询 SQLite 或 DuckDB
- 返回结构化结果
- 对空结果、错误结果和超时结果做友好处理

验收标准：

- 核心样例问题均可返回数据
- SQL 失败时不暴露原始错误堆栈

### F008 AI 总结结论

优先级：P0

需求：

- 根据查询结果生成业务化总结
- 结论需要包含关键发现、变化方向和可能原因
- 避免空泛表达

验收标准：

- 每个核心问题都有一段可读的业务结论

### F009 图表展示

优先级：P0

需求：

根据问题类型展示合适图表：

| 问题类型 | 图表 |
|---|---|
| 趋势类 | 折线图 |
| 排名类 | 柱状图 |
| 对比类 | 分组柱状图 |
| 占比类 | 环形图 |
| 时段类 | 小时热力图 |
| 异常类 | 表格 + 地图高亮 |
| 预测类 | 折线图 + 预测虚线 |

验收标准：

- 推荐问题至少能覆盖折线图、柱状图、表格和地图高亮

### F010 明细表格

优先级：P0

需求：

- 展示聚合后的查询结果
- 默认最多展示 20 行
- 金额、百分比和时间格式统一
- 不展示手机号、车牌等敏感字段

验收标准：

- 图表数据和表格数据一致

### F011 地图展示与联动

优先级：P0

需求：

- 地图展示上海都市圈示例充电站点
- 站点颜色表示状态
- 站点大小表示收入或订单规模
- 查询结果能高亮相关区域或站点
- 点击区域或站点后更新上下文

验收标准：

- 点击地图区域后，左侧能出现上下文提示
- 用户可以基于选中区域继续追问

### F012 多轮上下文对话

优先级：P0

需求：

系统需要维护：

- 时间范围
- 区域或站点
- 指标
- 上一轮结果集合
- 指代词

支持追问：

- 只看浦东呢？
- 再看收入
- 为什么下降？
- 这些站点故障多吗？
- 按周展示

验收标准：

- 至少支持 3 轮连续追问不断上下文

### F013 澄清交互

优先级：P1

需求：

当问题存在歧义时，系统主动澄清。

典型歧义：

- “表现最好”未说明指标
- “最近”未说明时间，但影响结果明显
- “利用率”可能是站点利用率或充电桩利用率

验收标准：

- 系统能给出 2 到 4 个可选口径按钮或追问文案

### F014 超范围问题处理

优先级：P0

需求：

用户询问非充电运营问题时：

- 说明当前产品支持范围
- 不直接胡答
- 推荐可问问题

验收标准：

- 对旅游、股票、路线规划等问题能稳定引导回产品范围

### F015 权限与隐私保护

优先级：P0

需求：

当用户查询手机号、车牌、个人明细时：

- 拒绝查询
- 解释原因
- 推荐聚合指标替代问题

验收标准：

- 任何敏感个人信息类问题都不会生成查询 SQL

### F016 简单预测

优先级：P1

需求：

- 使用近 30 天或近 8 周数据做趋势外推
- 支持未来 7 天订单、收入或高峰压力预测
- 标注预测为估算结果

验收标准：

- 能回答“未来 7 天哪些站点可能出现高峰压力？”

### F017 运营建议

优先级：P1

需求：

基于规则生成建议：

- 低利用率站点优化
- 高故障站点优先运维
- 高负荷站点扩容
- 节假日前巡检
- 凌晨低谷运营活动

验收标准：

- 建议需要关联具体数据原因，不输出泛泛建议

## 13. 成功指标

作品集 Demo 成功标准：

- 访客 5 秒内理解产品场景
- 访客 30 秒内完成一次推荐问题查询
- 核心样例问题查询成功率达到 90% 以上
- 至少支持 3 轮多轮追问
- 至少展示 4 种图表或可视化形态
- Schema 和 SQL 展示能体现可信解释
- 产品界面与个人网站视觉风格一致

## 14. 验收用例

操作：

1. 打开产品页

- 展示柱状图和表格

### 用例二：地图选区追问

操作：

1. 点击浦东新区
2. 输入“这个区域最近 30 天收入趋势如何？”

期望：

### 用例三：多轮指代

操作：

1. 问“上周故障率最高的 5 个站点是哪些？”

### 用例四：隐私拦截

操作：

1. 输入“查询某个用户的手机号和充电记录”

期望：

- 系统拒绝查询
- 不生成 SQL
- 推荐聚合替代问题

### 用例五：超范围引导

- 系统说明当前不支持路径规划
- 推荐充电运营相关问题

## 15. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 开放输入导致不稳定 | 用户体验差 | 推荐问题、SQL 白名单、模板兜底 |
| 地图实现复杂 | 工程成本高 | 先做轻量站点地图，复杂地图能力可降级 |
| 模拟数据不真实 | 作品可信度下降 | 明确数据说明，并参考公开数据结构 |
| LLM 生成错误 SQL | 查询失败或风险 | SQL 校验、只读限制、白名单表 |
| 多轮上下文错乱 | 追问体验差 | 明确保存时间、区域、指标和结果集合 |
| 视觉和个人网站割裂 | 作品集一致性下降 | 使用浅色 SaaS + 暖橙点缀 |
